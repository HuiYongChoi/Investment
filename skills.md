# Investment Navigator Working Skills

이 문서는 Investment Navigator에서 반복적으로 쓰는 **실전 작업 스킬/플레이북**을 한 곳에 정리한 운영 문서입니다.  
특히 PDF, 카카오, 라이브 배포처럼 한 번 삐끗하면 같은 증상을 여러 번 반복하기 쉬운 작업을 빠르게 재현하고 끝내는 데 목적이 있습니다.

## 1. PDF 리포트 검증 스킬

### 언제 쓰나
- PDF가 백지로 보일 때
- 흰 배경에 흰 글자가 나오거나, 아예 문서가 열리지 않을 때
- “코드는 맞는데 왜 안 보이지?” 같은 상황

### 최소 절차
1. 실제 앱에서 PDF 파일을 생성한다.
2. 생성 파일의 `파일 크기`, `페이지 수`를 확인한다.
3. `PyMuPDF(fitz)`로 첫 페이지를 이미지로 렌더링한다.
4. 렌더 결과 이미지에서 사람이 텍스트를 읽을 수 있는지 확인한다.
5. 그 전까지는 해결이라고 말하지 않는다.

### 권장 체크포인트
- PDF 헤더 `%PDF-1.x` 존재 여부
- 첫 페이지 렌더 이미지 존재 여부
- 생성 PDF가 브라우저/리더에서 실제로 열리는지
- 색상 반전/투명 배경/오프스크린 DOM 의존 여부

## 2. 카카오 메시지 디버깅 스킬

### 언제 쓰나
- `나에게 보내기`가 `401/403`으로 실패할 때
- 동의는 했는데 계속 alert만 뜰 때
- 설정은 다 맞는 것 같은데 코드가 의심될 때

### 최소 절차
1. 프론트 버튼 클릭 이벤트가 실제 어느 함수로 연결되는지 확인한다.
2. 브라우저 Network에서 요청 URL, 상태코드, 응답 body를 본다.
3. 프록시가 있다면 프록시 응답 `code/msg/status`를 확인한다.
4. Kakao SDK fallback/share 경로가 실제 실행됐는지 콘솔로 확인한다.
5. `설정 탓`으로 돌리기 전에, **프론트 → 프록시 → Kakao** 각 단계의 실패 지점을 분리한다.

### 금지사항
- 동일한 권한/리다이렉트 추정만 반복하지 않기
- 실제 응답 객체 없이 “권한 확인해달라”만 반복하지 않기

## 3. 라이브 배포 검증 스킬

### 언제 쓰나
- 서버에 반영했다고 말하기 전
- “내 로컬은 맞는데 서버는 이상함” 상황
- 캐시 때문에 UI가 과거 버전처럼 보일 때

### 기본 절차
1. `scp -> /home/bitnami -> backup -> sudo cp` 방식으로 배포한다.
2. `index.html`에서 JS/CSS `?v=`를 올린다.
3. `curl`로 라이브 HTML과 JS/CSS에 새 버전 문자열이 있는지 확인한다.
4. 핵심 함수명/문자열까지 `rg`로 확인한다.
5. 그 다음 실제 UI/산출물까지 본다.

## 4. 반복 실패 차단 스킬

### 언제 쓰나
- 같은 명령을 2번 이상 반복하려고 할 때
- 가설만 바뀌고 근거는 안 쌓일 때
- 사용자 피로도가 높아졌을 때

### 실행 규칙
1. 현재 실패 증거를 한 줄로 적는다.
2. 다음 수정이 어떤 가설을 검증하는지 한 줄로 적는다.
3. 수정 후엔 반드시 새 증거를 수집한다.
4. 새 증거가 없으면 다음 수정으로 넘어가지 않는다.

## 5. 현재 로컬에서 바로 쓸 수 있는 도구

이미 사용 가능한 도구:
- `PyMuPDF (fitz)` : PDF 첫 페이지 렌더링 확인
- `Pillow (PIL)` : 이미지 후처리
- `PyPDF2` : PDF 페이지 수, 텍스트 추출 확인

즉, **지금 당장 추가 설치가 꼭 필요한 상태는 아닙니다.**

## 6. 추가로 있으면 좋은 도구

선택 설치:
- `qpdf` : PDF 구조 검증
- `poppler` (`pdfinfo`, `pdftoppm`) : PDF 메타/렌더링 CLI

현재 이 로컬 환경에는 `brew`가 없어서 자동 설치는 하지 않았습니다.  
필요하면 나중에 Homebrew를 먼저 설치한 뒤 위 도구를 붙이는 방향이 좋습니다.
